Optimizer Data Processing Rules

These rules explain how the system prepares data, how OpenAI uses the prepared data, and how the backend validates the final allocation.

1. Data Sent to OpenAI
  • The system does not send the full church database to OpenAI.
  • Only summarized budget data is sent, such as monthly pool, ministry requirements, caps, priorities, and spending references.
  • Raw receipts, passwords, complete transaction tables, and unrelated personal data are not included in the optimizer payload.
  • The backend prepares the data first before OpenAI receives it.
2. Current Fund Mode Rules
  • Current Fund mode uses current unrestricted financial capacity.
  • The system starts with unrestricted cash available.
  • Projected unrestricted bills are reserved first before optimization.
  • The monthly pool is limited by the actual affordable amount and ministry need.
  • Old budgets from many years should not equally control Current Fund mode.
3. Historical Mode Rules
  • Historical mode uses the selected historical year as reference.
  • It considers historical unrestricted income and ministry spending activity from that year.
  • Historical mode is used for reference planning, not as the live cash position.
  • Historical recommendations may differ from Current Fund mode because the basis is different.
4. Emergency Expense Rules
  • The system does not rely on simple averaging alone.
  • Large abnormal expenses are treated as possible emergency spikes.
  • Emergency spikes are not automatically treated as normal monthly expenses.
  • The backend uses adjusted spending behavior before sending the need values to OpenAI.
5. Annual or Seasonal Expense Rules
  • If a ministry spends only once or twice a year, the system classifies it as annual or event-based spending.
  • Annual/event spending is converted into a monthly reserve amount.
  • Seasonal expenses are also spread as a monthly reserve.
  • This prevents one-time annual spending from being misread as monthly spending.
6. Inflation Rules
  • The backend provides the inflation context to OpenAI.
  • OpenAI must not guess or invent the inflation rate.
  • Inflation is applied only to budget needs, spending references, and historical references.
  • Inflation must not increase actual cash on hand, restricted funds, unrestricted cash, safe-to-spend amount, or monthly pool.
  • The final allocation is still limited by real available funds.
7. Allocation Rules
  • OpenAI must return allocations in monthly units only.
  • Do not allocate negative values.
  • Use every eligible ministry exactly once.
  • Do not exceed the provided monthly pool.
  • Respect ministry maximum caps when configured.
  • If funds are insufficient, higher-priority ministries may receive relatively more.
  • Any leftover amount is placed into reserve or unallocated balance.
8. Backend Validation Rules
  • OpenAI output is treated as a recommendation, not an automatic final decision.
  • The backend validates the returned allocations.
  • The backend prevents over-allocation beyond the available pool.
  • The backend prevents negative amounts and invalid ministry IDs.
  • The final plan is still reviewable by authorized church finance personnel.
Panel Defense Summary:
The optimizer uses processed and summarized church financial data. It handles emergency spikes, annual/event-based expenses, current fund limitations, and inflation-aware needs. OpenAI generates a recommendation from these constraints, while the backend validates the output to ensure the allocation does not exceed the actual available funds.
